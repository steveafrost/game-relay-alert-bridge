## App Overview

**game-relay-alert-bridge** is a minimal Vercel serverless function that bridges DigitalOcean Uptime Monitoring alerts to SMS via Twilio. It acts as a Slack webhook shim: DigitalOcean's Uptime checks can send Slack-format webhooks but not generic HTTP targets, so this function accepts Slack-shaped JSON payloads, extracts and compresses the alert text, and forwards it as an SMS message.

```
DigitalOcean Uptime alert → Vercel Function (Slack webhook shim) → Twilio SMS → phone
```

- **Language:** JavaScript (ESM, `node:test`)
- **Deployment target:** Vercel Serverless Functions (one endpoint)
- **Auth:** Shared secret via query param (`?token=...`) or `x-alert-secret` header
- **State:** Early-stage, single-commit initial public release

### Endpoint

**`GET /api/do-alert?token=<secret>`** — Health/config check. Returns `{ ok: true, configured: { ... } }` with booleans for each required env var.

**`POST /api/do-alert?token=<secret>`** — Main webhook receiver. Accepts Slack-shaped JSON body. Returns `{ ok: true, sid: "...", status: "..." }` on success, or error JSON on failure.

**All other methods** → `405 Method Not Allowed`.

### Data Flow (POST)

1. **Auth check** — `requireSharedSecret()` validates the shared secret from query param or `x-alert-secret` header.
2. **Payload extraction** — `readPayload()` reads the incoming body (object or JSON string).
3. **Message building** — `buildMessage()` orchestrates:
   - `summarizeSlackPayload()` — Extracts text from Slack-shaped JSON (`text`, `attachments[].title/text/fields`).
   - `normalizeAlertText()` — Strips Slack URL markup (`<url|label>`), emoji (`:emoji:`), extraneous whitespace.
   - `conciseDigitalOceanMessage()` — Pattern-matches DO-specific alert text and compresses known states:
     - **Down:** `"Relay down: <host> down globally for <duration>."`
     - **Recovered:** `"Relay recovered: <host> reachable again."`
     - **Unknown:** Falls back to generic `"Relay alert: <normalized text>"`.
   - Truncates to 1200 chars for SMS.
4. **Twilio send** — `sendTwilioSms()` sends via Twilio Messages API with HTTP Basic auth (API key/secret or account SID/auth token).

## Proactive Improvements

### 1. Alert Routing — Multiple Channels

The current single-SMS path is fragile. Add:

- **Channel abstraction** — Define an `AlertChannel` interface so new channels (email, Discord, Telegram) can be plugged in without changing the handler.
- **Configurable routing** — Allow env-var-based routing (e.g., `ALERT_CHANNELS=twilio,discord`).
- **Email fallback** — Use a mail service (SendGrid, Resend) as a secondary channel when SMS fails.
- **Discord/Telegram** — Simple webhook integrations for teams that don't want SMS-only alerts.

### 2. Alert Deduplication & Rate Limiting

DO may fire repeated "down" alerts. Without dedup, every alert becomes an SMS, which is expensive and annoying.

- **In-memory dedup cache** — Track alert hash by `firstDomain()` + state (down/recovered) with a TTL (e.g., 5 minutes). Since Vercel functions are stateless, use Upstash Redis or a simple KV store.
- **Cooldown** — Prevent sending more than 1 SMS per alert type per N minutes.
- **Rate limiting on endpoint** — Basic IP-based or token-based rate limiting to prevent abuse.

### 3. Monitoring & Observability

The function has zero observability:

- **Structured logging** — Use `pino` or a simple `console.log` wrapper with JSON output (Vercel captures stdout). Log every request: method, path, auth result, alert host, Twilio SID, response time.
- **Error tracking** — Integrate with Sentry or at minimum capture uncaught exceptions and promise rejections.
- **Metrics endpoint** — Add a `/api/metrics` endpoint (protected) exposing Prometheus-style counters (requests, alerts sent, failures).
- **Alert health check** — Add a periodic cron job that pings the health endpoint and alerts if it's down (meta-alerting).

### 4. Better Error Handling

- **Twilio retry** — Implement exponential backoff (3 retries) on Twilio API failures (529/429/5xx).
- **Input validation** — Validate Content-Type (`application/json`), check payload structure before processing, return `400` for malformed requests.
- **Timeout** — Add a configurable timeout to the `fetch()` call to Twilio (currently hangs on network blips).
- **Graceful degradation** — If Twilio credentials are invalid partway through a day, the health check should still report accurate state.
- **Catch-all error handler** — Wrap the handler body in try/catch so unhandled exceptions return `500` instead of crashing the Vercel process.

### 5. Configuration & Developer Experience

- **`vercel.json`** — Explicitly define the function config (memory, maxDuration, regions).
- **`.env.example`** — Document all env vars with placeholder values.
- **`.gitignore`** — Ignore `node_modules/`, `.vercel`, `.env`, `*.log`.
- **`npm run dev`** — Add a local dev script using `vercel dev` or a simple HTTP server.

### 6. Automated Suggestions (CI/Tests)

- **GitHub Actions CI** — Run `npm test` on push/PR. Add linting with `standard` or `biome`.
- **Add test cases**:
  - Auth: missing secret, wrong secret, header auth, query param auth.
  - Payload extraction: empty body, non-JSON body, malformed Slack payload.
  - `sendTwilioSms`: Mock the fetch call; test missing env vars, Twilio errors, success.
  - Handler: full request/response via `node:test` mock.
  - Edge cases: weird emoji, missing fields, HTML in text, very long payloads.
- **Type checking** — Add JSDoc annotations or migrate to TypeScript for a function this small.

### 7. Security

- **Content-Type enforcement** — Reject non-JSON POST bodies early.
- **Payload size limit** — Reject bodies > 64KB.
- **Header-based auth** — Already supported (`x-alert-secret`), good. Document it as primary (cleaner than query params which get logged).

## Summary

This is a tiny, well-intentioned project that solves a real problem (DO Uptime → SMS) with minimal code. The core logic is clean and the message compression is effective. However, as a production bridge carrying monitoring alerts, the lack of observability, error handling, deduplication, and multi-channel routing are significant gaps for a tool whose job is to *reliably deliver critical alerts*.

The highest-value improvements are: **logging**, **Twilio retry/timeout**, **test coverage**, and **CI/CD** — and these are all quick wins.
